# Lead Intake Phase-Two Opportunity Routing Log

Status: internal reusable delivery-readiness template
Last updated: 2026-06-14 17:02 CDT

## Purpose

Use this log after a Lead Intake Command System / Follow-Up Engine build, handoff, support adjustment, or 14-30 day review surfaces a possible next improvement. It keeps expansion ideas organized without turning them into unapproved scope, pricing, vendor choices, public proof, mailbox drafts, or production changes.

This is an internal operating artifact. It is not a client commitment, proposal, quote, support agreement, public proof approval, vendor decision, deployment action, auto-send approval, auto-write approval, or authorization to change a live workflow.

## When To Use This

Use this record when one of these signals appears:

- A client asks, "Can it also handle this other lead source?"
- The 14-30 day review identifies a recurring manual bottleneck.
- QA or support notes show repeated exception handling.
- A tracker view reveals a high-friction follow-up step.
- Proof capture finds a useful process improvement that should be productized.
- A prospect reply points toward a second workflow after the first intake map.

Do not use this log to store raw customer data, private message content, credentials, regulated data, or unapproved screenshots.

## Routing Principles

- Stabilize phase one before recommending phase two.
- Route ideas through evidence first, not enthusiasm.
- Keep client-facing language separate from internal hypotheses.
- Treat every expansion idea as discovery-needed until scope, owner, dependency, risk, and approval path are clear.
- Separate support fixes from true expansion work.
- Do not publish proof or quote impact from an opportunity unless the evidence and approval record support it.

| Route | Use When | Output |
| --- | --- | --- |

## Discovery Questions

Use only the questions that fit the opportunity.

1. Which exact source, status, owner, or follow-up step is creating the friction?
2. How often does it happen in a normal week?
3. What does the team do manually today?
4. What happens if the automation misses, duplicates, or delays this step?
5. Who owns the decision when there is an exception?
6. Which current tool must remain the source of truth?
7. Does this need to create internal reminders, reviewable drafts, or external messages?
8. What should stay manual by design?
9. What evidence would make this worth scoping as phase two?
10. What approval is needed before any build or public claim?

## Approval Gates

Do not move from this log into an external or production action without the relevant approval:

- Client-specific discovery outreach.
- Mailbox draft creation or send.
- Proposal creation for review, if it includes client-specific commitments.
- Final pricing or package pricing.
- Paid provider, CRM, hosted form, or tool selection.
- Environment variables, credentials, DNS, deployment, or launch-status changes.
- Auto-send, auto-reply, CRM write, sheet write, calendar write, or repository auto-write behavior.
- Public proof, screenshots, testimonials, client names, metrics, or case-study claims.
